**建議：** 在單一寫入、多讀者且資料不逾 100MB 的本地場景下，預設採用 **SQLite (WAL 模式)**。SQLite 提供成熟的 ACID 保證，寫入時自動記錄 WAL 日誌，於下次開啟時自動執行恢復 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=,the%20third%20connection%20will%20get))；WAL 模式允許多讀者與單一寫者併發執行，互不阻塞 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=1,and%20writing%20can%20proceed%20concurrently))。Python 標準庫內建 sqlite3，不需額外守護進程或重度依賴。雖然 JSONL 日誌與原子快照具有人可讀性優勢，但自行實作需處理截斷行、部分寫入等風險 ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text))。基於可靠性、效能與維護成本考量，SQLite/WAL 作為預設更合適。

**關鍵理由：** SQLite/WAL 的設計能自動保證崩潰安全與資料一致性（寫入時提交或回滾），避免自行編寫複雜的日誌回放機制。WAL 模式下，讀者與寫者並行且 I/O 多為順序存取 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=1,and%20writing%20can%20proceed%20concurrently))；在崩潰後，第一次開啟會觸發自動重播恢復 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=,the%20third%20connection%20will%20get))。SQLite 資料庫為單一跨平台檔案， ([www.sqlite.org](https://www.sqlite.org/onefile.html#:~:text=Single))支援拷貝或異機器使用，格式穩定。反觀 JSONL 日誌必須自行解析，每行 JSON 若因崩潰截斷損壞需透過掃描辨識並移除 ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text))；且在恢復時須先載入上次快照，再串流重放新增條目。若快照寫入時未使用原子替換，亦有殘留半成品檔風險，但可藉 POSIX `rename` 原子替換操作避免 ([man7.org](https://man7.org/linux/man-pages/man2/rename.2.html#:~:text=If%20newpath%20already%20exists%2C%20it,refer%20to%20the%20file%20being))。實踐上，如 Glacis 庫所示，SQLite 適合生產環境與查詢需求，JSONL 則適用簡單記錄和日誌管理 ([docs.glacis.io](https://docs.glacis.io/sdk/python/storage/#:~:text=SQLite%20%28default%29%20%20,friendly))。

**Failure modes：**
- *SQLite (WAL)*：如寫入過程中斷，WAL 檔案保留變更，下次打開時會重播並完成或回滾。若磁碟空間耗盡，最後一筆交易可能回滾到前一穩定狀態。WAL 可能成長過大（未檢查點時），長交易（>數百MB）在極端情況下效率不佳 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=8.%20,with%20large%20transactions%20as%20does))；此外，WAL 僅保證在本地檔案系統有效，不支持跨網路磁碟。SQLITE 檔案格式穩定，但若使用極舊版本可能無法讀取 WAL 模式資料庫 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=The%20database%20file%20format%20is,database%20that%20is%20operating%20in))。
- *JSONL 日誌＋快照*：資料以 NDJSON 行形式追加，易於以文本檢視、版本控制；但若寫入途中截斷，會遺留不完整的 JSON 行。必須實作掃描邏輯，丟棄非完整記錄 ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text))。快照更新需寫入臨時檔並以 `rename` 原子移動，才能避免中間狀態 ([man7.org](https://man7.org/linux/man-pages/man2/rename.2.html#:~:text=If%20newpath%20already%20exists%2C%20it,refer%20to%20the%20file%20being))。如果快照和日誌同步處理不當，可能導致遺失最新更新或重複應用部份日誌。恢復流程需先載入最後一次快照，再順序重放日誌中自上次快照後的條目。為節省空間，需定期「壓縮」日誌：刪除已合併至快照的舊條目，並以原子方式重寫檔案確保 crash-safe ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=Compaction%20guarantees%3A%20,safe%20behavior))。

**切換條件：** 若有強烈需求則可擇 JSONL。典型例子包括需要將整個日誌目錄放入 Git 管理、便於人為審核或異機種間簡易複製（只需傳送文本檔案），此時 JSONL 設計較為友好 ([docs.glacis.io](https://docs.glacis.io/sdk/python/storage/#:~:text=SQLite%20%28default%29%20%20,friendly)) ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text))。另外，如果資料模型極其簡單、主要以追加記錄方式使用，且刻意希望所有變更可「明文查閱」，JSONL 可考慮採用。反之，若將來需要複雜查詢、索引或多版本管理，或預期資料量超過百 MB，則 SQLite 將更具優勢。不符合單機條件（如部署在網路檔系統上）也會打破 WAL 的假設，此時需考慮 JSONL 或其他方式。

**可逆下一步：** 先建立一個抽象化的儲存接口，將資料存取（事件追加、狀態重建）與底層實作解耦。例如參考 Glacis SDK 的做法，撰寫一個通用 Storage 類，根據配置使用 SQLite 或 JSONL。這樣可按需要增加 JSONL 後端，並提供互轉工具：例如寫腳本根據事件日誌初始化或更新 SQLite 資料庫，或將 SQLite 資料導出為 JSONL。確保數據結構或序列化格式在兩者間一致，以便隨時切換。實作階段可先以 SQLite 為主，再逐步驗證將現有資料匯出至 JSONL 並恢復，確保方案具有可逆性。

**Acceptance tests：**
- 使用 Python `sqlite3` 模組開啟 WAL 模式的資料庫，執行多筆 `INSERT/UPDATE` 且呼叫 `commit()`，關閉後重新開啟，檢查所有已提交的資料皆存在（無遺失或重複）。
- 建立兩個獨立連線，一個持續寫入多筆資料，另一個同時進行讀取查詢。確保在 WAL 模式下讀者不會因寫者而阻塞（查詢可正常完成，未拋出 SQLITE_BUSY） ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=1,and%20writing%20can%20proceed%20concurrently))。
- 模擬崩潰：在寫入資料前故意觸發例外（或`os._exit(1)`），導致當前事務未提交。再次啟動程式並開啟資料庫，檢查該事務未引入半成品，資料庫保持上一個穩定狀態（WAL 邏輯應已將不完整事務回滾）。
- 測試 JSONL 方案時：向日誌檔 append 正確格式的 JSON 行數筆，在不關閉檔案情況下使程式異常終止，然後啟動恢復程序，檢查損壞行是否被安全移除，其餘完整插入是否正確讀回 ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text)) ([man7.org](https://man7.org/linux/man-pages/man2/rename.2.html#:~:text=If%20newpath%20already%20exists%2C%20it,refer%20to%20the%20file%20being))。
- 驗證快照策略：先以 SQLite 或 JSON 寫出狀態快照到臨時檔，再以原子 `rename` 交換新檔名。測試任意中斷快照寫入（例如提早 `exit`），重啟系統後應只能看到完全寫入的舊版本或新版本，永遠不會讀到半成品 ([man7.org](https://man7.org/linux/man-pages/man2/rename.2.html#:~:text=If%20newpath%20already%20exists%2C%20it,refer%20to%20the%20file%20being))。

**Sources：** 官方 SQLite 文檔指出 WAL 模式下讀寫並行、性能良好，並能自動在首次打開時重播上次崩潰後的日誌 ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=1,and%20writing%20can%20proceed%20concurrently)) ([www.sqlite.org](https://www.sqlite.org/wal.html#:~:text=,the%20third%20connection%20will%20get))；POSIX `rename` 系統調用保證原子替換檔案 ([man7.org](https://man7.org/linux/man-pages/man2/rename.2.html#:~:text=If%20newpath%20already%20exists%2C%20it,refer%20to%20the%20file%20being))；Papyra 等工業標準描述了 JSONL append-only 日誌的設計原則，強調需顯式處理截斷行以達成崩潰安全 ([papyra.dymmond.com](https://papyra.dymmond.com/persistence/json-backend/#:~:text=2,or%20text))；Glacis 官方文檔也展示了 SQLite 與純 JSONL 儲存各自適用場景 ([docs.glacis.io](https://docs.glacis.io/sdk/python/storage/#:~:text=SQLite%20%28default%29%20%20,friendly))。以上來源皆支持上述結論和設計選擇。
